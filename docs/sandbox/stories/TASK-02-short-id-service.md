# TASK-02 — Short ID Generation Service

**Type:** Technical Task  
**Phase:** 0 — Foundation  
**Estimate:** XS

## Goal

Implement a short, URL-safe, collision-resistant ID generator used as the primary key for snippets.

## Acceptance Criteria

- [ ] IDs are **8 characters** long, using the alphanumeric character set `[a-zA-Z0-9]` (62 chars → ~218 trillion combinations; negligible collision risk at v1 scale).
- [ ] IDs are generated by the API service at snippet creation time — not by the database.
- [ ] On collision (extremely rare), the service retries with a new ID up to 3 times before returning a 500.
- [ ] The generator is a small utility function (not a separate service) injectable via NestJS DI or importable directly.
- [ ] Unit tests cover: output length, character set, and basic collision-retry logic.

## Implementation

```ts
import { customAlphabet } from 'nanoid';

const ALPHABET = 'ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789';
const generateId = customAlphabet(ALPHABET, 8);

// In SnippetService.create():
const MAX_RETRIES = 3;
for (let i = 0; i < MAX_RETRIES; i++) {
  const id = generateId();
  const existing = await prisma.snippet.findUnique({ where: { id } });
  if (!existing) return id;
}
throw new InternalServerErrorException('Failed to generate unique snippet ID');
```

## Notes

- Use the `nanoid` package (already widely used in the ecosystem; small bundle, cryptographically random).
- 8 characters at 62-char alphabet = 62^8 ≈ 2.18 × 10^14 combinations. Even at 1 M snippets the collision probability per generation is ~4.6 × 10^-9 — the retry loop is a safety net only.
- The `findUnique` check on insert is a simple guard; for high-scale scenarios a unique constraint violation catch-and-retry is more robust, but the check is sufficient for v1.
